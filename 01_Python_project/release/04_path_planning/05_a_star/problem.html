A* — heuristic 기반 8-connected 최단 경로

Chapter 4 · Path Planning / 05_a_star  |  Dijkstra 에 방향성을 더해 탐색을 줄인다

A* 알고리즘으로 격자 미로에서 start 부터 goal 까지 최단 경로를 탐색하세요. Dijkstra(04)와 같은 8-connected 격자이지만, heuristic 을 도입해 expand 횟수를 줄입니다.
이 과제는 차량 제어가 없습니다. 결과는 Rerun 2D viewer 의 노드 expansion 과 최종 경로로만 확인합니다.

1문제 상황

Dijkstra 는 시작점부터의 누적 cost g 만 보고 모든 방향으로 균등하게 확장합니다 — goal 이 어디 있든 상관없이 퍼지죠. A* 는 거기에 goal 까지 남은 거리 추정치(heuristic h)를 더해, goal 방향을 우선 확장하므로 같은 답을 더 적은 expand 로 찾습니다.

Dijkstra (04)A* (05)
우선순위f = gf = g + weight · h
확장 패턴모든 방향 균등goal 방향 우선
  • weight_heuristic = 1.0 (admissible) — 최단 경로를 보장합니다.
  • weight_heuristic > 1 — greedy bias. expand 수는 줄지만 최적성은 약해집니다.

2Map 구조 map_2

  • 외곽 border + 중앙 vertical wall(x=30, y=10..50).
  • box 의 상·하 면(y=10 / y=50, x=20..40) — 좌·우 면(x=20 / x=40)은 개방.
  • start (10, 30), goal (50, 30) — 둘 다 box 밖, 중앙 벽 우회 필수.

3구현 과제

a_star.py 의 a_star() 함수 본문(# TODO)을 구현하세요. 시그니처는 이미 주어져 있습니다. 

def a_star(
    start: tuple[int, int],
    goal: tuple[int, int],
    obstacles: set[tuple[int, int]],
    weight_heuristic: float = 1.0,
    on_step: Callable[[tuple[int, int], set[tuple[int, int]], float, float], None] | None = None,
) -> list[tuple[int, int]]
인자 의미
obstacles 벽 셀의 집합 — O(1) 포함 검사용 set.
weight_heuristic heuristic 가중치(기본 1.0). f = g + weight · h 의 weight.
on_step 매 노드 expand 직후 호출되는 시각화 콜백 — (current, open_set, g_cost, f_cost). open_set 은 그 시점의 open list, g_cost 는 current 의 누적 비용, f_cost = g + weight·h. None 이면 호출 안 함.
반환 경로 list(start → goal). 미발견 시 [].

거리 계산용 _euclid(a, b) 도우미가 a_star.py 에 이미 정의돼 있습니다.

게인·파라미터 튜닝

이 과제에서 구현하는 것은 알고리즘 로직 입니다. 튜닝할 수 있는 값은 weight_heuristic 하나로, record_gen.py 의 --weight 옵션으로 viewer 에서 자유롭게 실험할 수 있습니다 (예: --weight 10 = greedy). 합격 기준 검증의 weight=1.0 은 test_*.py 에 박혀 있으며 변경 금지 입니다 — admissible 한 1.0 에서만 최단성을 보장하기 때문입니다.

디버그 신호 — expansion 별 분석 (선택)

record_gen.py 의 on_step 콜백은 매 노드 확장(expansion)마다 호출됩니다 — 거기서 goal_dist(현재 노드→goal 거리)·open_size (open list 크기)·g_cost(시작→current 누적 비용)·h_cost(heuristic 항 weight·h)·f_cost(총 비용 g + weight·h)를 record 의 debug_scalars 로 저장합니다. 더 분석할 값은 dbg.add(...) 의 kwarg 한 줄로 추가할 수 있습니다.

디버그 신호는 viewer 기본 화면에는 없습니다 — 좌측 entity 패널에서 /debug/<이름> 을 골라 TimeSeriesView 를 추가하세요. 하단 타임라인은 기본이 iteration (expansion 단위)이라 디버그 곡선이 expansion 축으로 곧바로 보입니다. frame 단위 (--skip 묶음)로 재생하려면 타임라인 드롭다운에서 step 을 고르세요.

A* 의 goal_dist 를 Dijkstra(04)와 나란히 비교하면 heuristic 효과가 한눈에 보입니다 — Dijkstra 는 front 가 사방으로 퍼져 들쭉날쭉하고, A* 는 goal 쪽으로 곧장 줄어듭니다.

5합격 기준 pytest

알고리즘 형태(정통 A* / heapq priority queue / 다른 변종)는 자유 — 동작 명세(behavioral spec)만 검증합니다.

  1. 시작·끝 일치 — path[0] == start, path[-1] == goal.
  2. 장애물 회피 — 모든 path 노드가 obstacles 밖.
  3. 8-connectivity — 연속 노드 간 이동이 8방향 중 하나.
  4. admissible 최단성 — weight_heuristic=1.0 에서 path cost 가 reference 의 1.05배 이내.
  5. on_step 콜백 호출 — 첫 호출의 current 가 start, 호출 횟수 ≥ path 길이.
주의: heuristic 을 잘못 적용해 발산하거나 경로를 찾지 못하는 구현은 임계값 초과로 차단됩니다.

6힌트

  • Dijkstra 와의 차이는 우선순위 하나 — open 에서 노드를 고를 때 누적 cost g 만 보는 게 Dijkstra, g + weight · h 를 보는 게 A* 입니다. 나머지 루프 구조(open/closed, 이웃 확장, 경로 복원)는 04 와 똑같습니다.
  • heuristic h 는 현재 노드에서 goal 까지의 직선거리(_euclid)로 잡습니다 — 실제 거리를 넘지 않으므로(admissible) 최단성이 보장됩니다.
  • 이웃을 확장할 때 자식의 g 는 부모의 g + 이동 비용, 자식의 f 는 그 g 에 weight · h 를 더한 값입니다.
  • open 이 비면 경로 미발견 — 빈 list 를 반환합니다. 경로 복원 후에는 start → goal 순서로 뒤집어 반환하세요.
