Hybrid A* — kinematic 경로 탐색 + Pure Pursuit 추종

Chapter 4 · Path Planning / 06_hybrid_a_star  |  차량이 실제로 달릴 수 있는 경로를 탐색

Hybrid A* 로 kinematic 차량의 연속 상태 (x, y, yaw) 공간에서 경로를 탐색하고, 결과 path 를 Chapter 3 의 PurePursuit 으로 따라가 보세요.

1문제 상황

A*(05)가 찾은 격자 경로는 대각선·직각으로 꺾입니다 — 사람이 걷는 길이지 차가 달릴 수 있는 길이 아닙니다. Hybrid A* 는 노드를 격자 칸이 아니라 차량 운동학으로 도달 가능한 pose 로 확장하므로, 결과 경로가 곧바로 주행 가능한 곡선이 됩니다.

2Map 구조 map_3

  • 탐색 공간: x ∈ [-2, 15], y ∈ [-2, 15].
  • start (10, 0, π) — 좌측을 바라봄. goal (10, 10, 0) — 우측을 바라봄.
  • 장애물: 반지름 1 의 원형 17개 — vertical wall(x=3, y=5..10) + horizontal wall (x=4..14, y=5)이 L-shape 를 이룹니다. 좌측 위로 우회해야 합니다.

4구현 과제

hybrid_a_star.py 의 hybrid_a_star() 함수 본문(# TODO)을 구현하세요. 시그니처는 이미 주어져 있습니다. 

def hybrid_a_star(
    start: tuple[float, float, float],
    goal: tuple[float, float, float],
    space: tuple[float, float, float, float],
    obstacles: list[tuple[float, float, float]],
    R: float = 5.0, vx: float = 2.0, dt: float = 0.5,
    weight: float = 1.1, epsilon_goal: float = 0.6,
    vehicle_radius: float = 1.7,
    on_step: Callable[[pose, pose | None, set[key], float, float], None] | None = None,
) -> list[tuple[float, float, float]]
인자의미
space(x_min, x_max, y_min, y_max) 탐색 공간.
obstacles(x, y, radius) 튜플 리스트 — 원형 충돌.
vehicle_radius차량 effective 반경 + 안전계수. arc_collision 에서 장애물 반경에 Minkowski-inflation 으로 더해짐.
on_step매 expand 직후 호출 — (current_pose, parent_pose_or_None, open_set, g_cost, f_cost). start expand 시 parent 는 None. open_set 은 그 시점의 open list, g_cost 는 누적 비용, f_cost = g + weight·h.
반환pose list(start → goal). 미발견 시 [].

게인·파라미터 튜닝

이 과제에서 구현하는 것은 알고리즘 로직 입니다 — 함수는 시그니처가 고정이고 본문만 채웁니다. 탐색 동작을 좌우하는 파라미터(R, vx, dt, weight)는 record_gen.py 의 module-level 상수로 정의돼 있어, 값을 바꿔 다시 실행하면 search 결과·속도가 달라지는 것을 viewer 에서 비교할 수 있습니다 (자유롭게 변경 OK). 합격 기준 검증용 값은 test_*.py 에 있으며 변경 금지 입니다.

구현 파일

시나리오 실행 → record JSON 생성 + Rerun viewer 자동 실행

기본 모드 — search phase 만 (알고리즘 디버깅용):

uv run python 01_Python_project/release/04_path_planning/06_hybrid_a_star/record_gen.py

2D viewer 에 탐색 공간 + 원형 장애물 + start(파랑) + goal(빨강)이 표시되고, 하단 iteration 타임라인 scrubber 로 expansion arc 가 트리처럼 자라는 과정을 재생합니다. 04·05 와 같은 simulator_search.py 뷰어라 디버그 신호도 같은 expansion 축에서 분석할 수 있습니다.

--controller 모드 — 추종까지 (최종 비주얼라이즈용):

uv run python 01_Python_project/release/04_path_planning/06_hybrid_a_star/record_gen.py --controller

3D viewer 에 반투명 회색 원기둥(장애물) + 지면 경계선 + ego 차량 + 빨간 goal marker 가 표시됩니다. timeline sim_time scrubber 로:

  • t = 0 ~ T_search — ego 는 START 에 정지. 회색 search arc 들이 batch 단위로 트리처럼 자라납니다. 우측 텍스트 패널에 closed / open / current pose.
  • t = T_search — 핑크 곡선(최종 path) 등장.
  • t > T_search — ego 가 Pure Pursuit 으로 path 를 매끄럽게 따라가 goal 에 도착.

두 모드 공통: JSON 만 만들려면 --no-viewer, search arc batch 를 묶으려면 --skip N(기본 20).

Rerun viewer 재생

uv run python 01_Python_project/release/04_path_planning/simulator_search.py 01_Python_project/release/04_path_planning/06_hybrid_a_star/

위는 기본(search) record 용입니다. --controller 로 만든 record 는 simulator_path_planning.py 로 재생하세요.

디버그 신호 — expansion 별 분석 (선택)

record_gen.py 의 on_step 콜백은 매 노드 확장(expansion)마다 호출됩니다 — 거기서 goal_dist(현재 pose→goal 거리)·open_size (open list 크기)·g_cost(시작→current 누적 비용)·h_cost(heuristic 항 weight·h)·f_cost(총 비용 g + weight·h)를 record 의 debug_scalars 로 저장합니다. 더 분석할 값은 dbg.add(...) 의 kwarg 한 줄로 추가할 수 있습니다.

디버그 신호는 viewer 기본 화면에는 없습니다 — 좌측 entity 패널에서 /debug/<이름> 을 골라 TimeSeriesView 를 추가하면 됩니다. 기본 모드는 하단 타임라인이 iteration(expansion 단위)이라 디버그 곡선이 곧바로 expansion 축으로 보입니다(04·05 와 동일). --controller 모드는 재생 타임라인이 연속 sim_time 이라, 디버그 곡선의 x축(expansion 인덱스 0, 1, 2, …)이 재생축과는 별개입니다.

6합격 기준 pytest

알고리즘 형태는 자유 — 동작 명세(behavioral spec)만 검증합니다.

  1. goal 도달 — path 의 마지막 pose 가 goal 에서 epsilon_goal=0.6 이내.
  2. start 일치 — path[0] 가 start 와 0.3 m 이내.
  3. 장애물 회피 — 모든 path pose 가 effective 반경(orad + vehicle_radius, 기본 1.7 m) 밖.
  4. kinematic 일관성 — 연속 pose 간 거리 ≤ vx·dt·1.05(motion primitive 이동 거리).
  5. on_step 콜백 호출 — 호출 수 ≥ path 길이, 첫 호출 pose ≈ start, 첫 호출 parent_pose 는 None.

7힌트

  • 자료 구조 — open/closed dict 의 key 는 좌표가 아니라 discretize_pose(pose) 가 만든 양자화된 bucket key 입니다. 연속 pose 라서 같은 위치를 정수로 묶어야 중복 확장을 막을 수 있습니다.
  • 핵심 루프는 A*(05)와 같은 구조 입니다 — open 에서 f 최소 노드를 골라 closed 로 옮기고, on_step 을 호출한 뒤, goal 도달이면 부모 체인으로 path 복원, 아니면 이웃을 확장. 다른 점은 이웃이 8방향 격자 이동이 아니라 motion_primitives 5개 원호라는 것뿐입니다.
  • 이웃 확장 — 각 motion primitive 에 대해 vehicle_move 로 child pose 를 계산하고, in_space + arc_collision(..., vehicle_radius=vehicle_radius) 로 안전성을 검사합니다.
  • on_step 의 parent_pose 는 closed 에 저장해 둔 부모 pose 입니다 — start 노드는 부모가 없으므로 None.
